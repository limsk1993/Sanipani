개발자 테스트 케이스 작성

 - 제품이 만들어지고 난 뒤 요구된 기능들이 제대로 작동하는지 확인하는 활동

 - 폭포수 개발 모델 : 순차적

 - 소프트웨어 테스팅 : 검증(절차) / 확인(결과)

 - 완벽한 테스팅은 불가능하다. (고객의 요구사항에 맞춤)

 - 소프트웨어 테스트 기법

블랙박스 기법(코드가 안보이는 상태, 사용자) / 화이트박스 기법(코드가 보이는 상태, 개발자)

 * 기준
블랙박스 기법 : What (무엇을 수행하는가)
화이트박스 기법 : How (어떻게 처리되는가)

 - 경험 기반 비법 (이 기법으로 시나리오 작성)
경험기반 -> 요구정의서
체크리스트(테스트의 기본)

 - 테스트 케이스 : 입력값을 주고 그 실제 출력값과 예상 출력값과 같은지 비교

 - 테스트 케이스 명세서 공통 부분
변경 기록 : 문서 변경 기록 및 변경 사유, 수정자, 역할 등 기술
승인 : 명세서의 승인자, 승인 관련자
범위 : 테스트 대상 소프트웨어 제품이나 시스템 아이템 및 기능 요약
담당 조직 : 테스트 케이스 명세서의 작성 및 배포 담당 조직

 - 테스트 케이스 명세서 개요
테스트 환경 요구사항 => 사전 사항, 사전에 뭐가 필요한지
ex) 글쓰려면 로그인 필요

 * 단계별 테스트 절차

단위 테스트 -> 통합 테스트 -> 시스템 테스트 -> 운영 테스트

단위 테스트 : 모듈로 분할하여 모둘별로 테스트

통합 테스트 : 모듈들을 합친 상태의 테스트

시스템 테스트 : 만든 프로그램과 기존의 프로그램을 같이 테스트

운영 테스트 : 실제

(통합, 시스템) 혹은 (시스템, 운영) 테스트가 묶이기도 함

 - 테스트 시나리오 작성 시 고객의 요구사항을 최우선

 *** 시험 : 고객의 입장이 되어 단위테스트의 블랙박스 형태로 시험

 - 실무 테스트 시나리오 작성

ex1)
테스트 시나리오 ID
테스트 시나리오 (제목)
테스트 시나리오 상세 설명 (테스트 요약)

ex2)
테스트 시나리오 ID
테스트 시나리오 (제목)
테스트 케이스 ID
테스트 케이스 설명 (상황)
사전 조건
테스트 데이터
예상 결과
결과 확인 방법
실제 결과
Pass / Fail
 * 실제 결과와 Pass / Fail은 실제 고객이 체크
